# v0.3.45 communications deployment runbook

This runbook activates the consent-aware Brevo integration after the application code is deployed. The code, database migration, and provider configuration are separate on purpose. A source deployment cannot send mail by itself.

## Safety state

- New templates are disabled and have no provider template ID.
- The database emergency stop begins in the paused state.
- Optional marketing begins separately paused, even after service delivery is resumed.
- A missing provider key causes safe queue deferral.
- Test mode sends only to the protected server allowlist.
- Consent, suppression, verified ownership, email hash, frequency, quiet hours, and quota are checked again after a worker claims each message.
- Receipt content, raw print data, activation keys, credentials, logs, email-address template fields, and private-network addresses are rejected from template parameters.

## 1. Prepare Brevo

1. Create a dedicated Brevo REST API key for POS Printer Emulator. Do not reuse the SMTP password.
2. Add and verify `posprinteremulator.com` as a sending domain.
3. Publish the account-specific Brevo authentication and DKIM records and a reviewed DMARC policy in IONOS DNS.
4. Verify `support@posprinteremulator.com` as the sender.
5. Create the reviewed transactional templates. Keep activation values and sensitive diagnostic data out of every template.
6. Record each numeric Brevo template ID. Do not put IDs or API keys in source code.

## 2. Configure protected delivery

Set the following environment variables only in the deployment process:

```text
PPE_BREVO_API_KEY
PPE_BREVO_SENDER_EMAIL
PPE_BREVO_SENDER_NAME
PPE_BREVO_REPLY_TO_EMAIL
PPE_BREVO_MODE=test
PPE_BREVO_TEST_ALLOWLIST
```

Use the existing SFTP variables and run:

```text
website-publisher configure-communications /admin_posprinteremulator
```

The C# publisher:

- validates the values;
- creates or updates the Brevo transactional webhook with bearer authentication;
- uploads `private/communications.php`;
- uploads a CLI-only `private/communications-cron.php`;
- stores webhook recovery metadata using Windows current-user data protection; and
- never displays the API key or webhook token.

## 3. Apply the additive database migration

```text
website-publisher migrate-communications https://admin.posprinteremulator.com/api/v1/migrate-communications.php
```

The migration is idempotent. It creates the outbox, templates, campaigns, delivery events, quota, and consented aggregate lifecycle tables. It does not enable a template or resume delivery.

## 4. Configure IONOS scheduling

Create an IONOS scheduled task that runs the protected CLI script every five minutes:

```text
php /absolute/account/path/admin_posprinteremulator/private/communications-cron.php
```

Use the absolute server path shown by IONOS. The script rejects web execution, schedules idempotent maintenance reminders, and processes at most 50 ready messages per run.

## 5. Controlled verification

1. Open **Admin Portal → Communications**.
2. Leave delivery paused.
3. Map only one reviewed Brevo template ID and enable it.
4. Confirm the intended test recipient is verified and included in `PPE_BREVO_TEST_ALLOWLIST`.
5. Queue a controlled test.
6. Resume delivery, let the worker process the test, and pause again.
7. Confirm one Brevo delivery, one provider message ID, and one deduplicated webhook event.
8. Send hard-bounce and unsubscribe provider tests. Confirm suppression prevents another eligible send.
9. Queue more than 300 simulated messages. Confirm 240 nonessential messages use the general automated allowance, 50 essential messages use the reserved service allowance, the remaining messages defer, and 10 provider slots remain for reviewed essential service delivery.

## 6. Production activation

After sender authentication, policy, quota, suppression, webhook, and test evidence pass:

1. Re-run protected configuration with `PPE_BREVO_MODE=live`.
2. Enable reviewed templates one at a time.
3. Resume delivery in the Admin Portal with a recorded business reason.
4. Monitor quota, deferred, failed, unknown-delivery, bounce, complaint, and unsubscribe states daily during the initial rollout.

Never retry a `DeliveryUnknown` message without checking Brevo first. A network interruption can occur after Brevo accepts a message but before the application receives the response.
