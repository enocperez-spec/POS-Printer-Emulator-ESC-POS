# v0.3.45 Brevo email integration design

## Decision

POS Printer Emulator will use Brevo as the external email-delivery platform for the consent-aware lifecycle communications planned for v0.3.45. The POS Printer Emulator administrative database remains the system of record for customer identity, license ownership, maintenance, consent, suppression, message intent, and audit history. Brevo receives only the minimum data required to deliver and report on a message.

The first production configuration targets the Brevo Free plan. The plan currently permits 300 email sends per day, does not roll unused sends into the next day, includes Brevo branding, and provides basic reporting. These limits are configuration rather than hard-coded product assumptions so the integration can move to a paid Brevo plan without a software release.

## Integration boundaries

- Only the protected backend communicates with `https://api.brevo.com`.
- The Admin Portal, Customer Portal, public purchase site, and desktop application never receive the Brevo API key.
- Required service messages and optional marketing messages use separate internal categories, consent rules, templates, metrics, and queue priorities even when they share one Brevo account.
- The administrative database owns the durable outbox. Brevo is the delivery provider, not the authoritative CRM or entitlement database.
- Receipt text, raw print data, screenshots, diagnostic attachments, activation keys, credentials, full IP addresses, and private listener details are prohibited from Brevo contacts, templates, tags, events, and webhook logs.

## Transport choice

- The Brevo REST API remains the primary production transport because it returns provider message identifiers and supports the template, contact, event, and webhook workflows required by this release.
- Do not use the deprecated `sendinblue/APIv3-php-library` or the legacy `Mailin.php` example. If the production backend uses a PHP SDK, install the current official `getbrevo/brevo-php` package through Composer, pin and audit the selected supported major version, use its typed transactional-email client, and read `BREVO_API_KEY` only from protected server configuration.
- The Brevo SMTP submission endpoint `smtp-relay.brevo.com:587` with STARTTLS is retained as a protected fallback for narrowly scoped transactional delivery or operational recovery. SMTP credentials are not an API key and do not replace the REST API or webhook credentials.
- Development SMTP credentials are stored outside the repository in a Windows DPAPI-encrypted, current-user-only secret file. Production credentials will be loaded from protected server configuration outside the web root and rotated before use.
- Neither SMTP nor REST credentials may appear in PHP source, JavaScript, desktop binaries, environment diagnostics, support exports, Git history, or deployment output.

## Delivery flow

1. A verified application event creates one immutable message intent with a unique idempotency key.
2. The policy evaluator confirms message category, recipient ownership, verified email, consent or service basis, suppression state, template version, frequency cap, and send time.
3. An eligible intent enters the durable outbox with a priority and next-attempt time.
4. A scheduled worker claims a bounded batch transactionally and checks the local daily quota ledger.
5. The backend calls Brevo `POST /v3/smtp/email` with a verified sender, a Brevo template ID, minimal template parameters, an internal non-secret reference, and a category tag.
6. The Brevo `messageId` is stored against the outbox record. A network-uncertain submission is reconciled before retry so a timeout does not automatically create a duplicate message.
7. Authenticated Brevo webhooks update sent, delivered, deferred, opened, clicked, bounced, complaint, blocked, error, and unsubscribe state idempotently.
8. Hard bounce, invalid address, complaint, and unsubscribe events update the appropriate suppression record before another optional message can be sent.

## Free-plan quota policy

The queue must not rely on Brevo accepting or retaining over-quota mail.

- `provider_daily_limit` defaults to 300 and is configurable in protected Admin settings.
- `automated_daily_limit` defaults to 290, leaving ten sends for controlled testing or emergency manual service mail.
- Fifty of the automated sends are reserved for security and essential service messages. The remaining 240 are available to normal service and consented marketing messages.
- Priority order is: account verification and password recovery; purchase, activation, renewal, and entitlement notices; support confirmations and replies; imminent maintenance or promotional-trial expiration; onboarding and setup guidance; release and upgrade education; inactivity and re-engagement marketing.
- Lower-priority mail remains queued for the next provider quota window. It is not dropped, duplicated, or silently marked as sent.
- Marketing campaigns are expanded into individual internal message intents in bounded batches, so a large audience cannot consume reserved service capacity.
- The Admin Portal shows used, reserved, available, queued, deferred, failed, and oldest-waiting counts plus the provider-account time zone used for the daily reset.
- A provider `429`, temporary failure, or depleted quota applies bounded exponential backoff with jitter. Permanent recipient failures are not retried.
- Administrators can pause all sends, pause marketing only, cancel unsent campaign messages, and perform a quota-safe test send.

## Message categories

### Required service messages

- Email verification, sign-in security, password reset, and account recovery.
- Purchase confirmation, activation delivery, license or computer changes, maintenance renewal, refund, and chargeback status.
- Promotional-trial activation and expiration where needed to explain an entitlement change.
- Support-request receipt, status, and staff response.
- Legally or operationally necessary account and service notices.

These messages do not depend on optional marketing consent, but they must still be expected, proportionate, privacy-safe, and documented in the customer notice.

### Optional marketing messages

- Product education that is not necessary to operate a requested service.
- Upgrade recommendations, release announcements, inactivity campaigns, and promotional offers.
- Conversion and re-engagement campaigns.

These messages require current affirmative marketing consent. An unsubscribe, complaint, global suppression, account closure, or withdrawn consent blocks them before queueing and again immediately before delivery.

## Brevo contacts and templates

- Sync only verified email, preferred display name, internal opaque customer reference, locale, and current marketing-consent state when the feature needs a Brevo contact.
- Do not synchronize license keys, installation IDs, payment-provider IDs, support descriptions, receipt activity details, or private telemetry.
- Templates are created and reviewed in Brevo, while the application stores the approved Brevo template ID, internal purpose, category, locale, version, required parameters, enabled state, and test evidence.
- Template parameters use allowlists per template. Arbitrary database fields or administrator-authored HTML cannot be injected into a send request.
- Every marketing template includes the required unsubscribe mechanism and business identification. Plain-text alternatives are required for all customer messages.

## Webhooks and reconciliation

- Create separate production and test webhook endpoints over HTTPS.
- Configure Brevo bearer-token webhook authorization and optionally allowlist current Brevo webhook IP ranges as defense in depth.
- Store the webhook credential outside the web root and rotate it independently from the API key.
- Reject missing or incorrect authorization, unexpected methods and content types, oversized payloads, malformed JSON, unknown event types, and timestamps outside the accepted reconciliation rules.
- Make webhook processing idempotent using the provider message ID, event type, event timestamp, and internal reference. Preserve the original event receipt only as long as needed for audit and reconciliation, with sensitive fields redacted from logs.
- Prefer webhooks over polling for delivery statistics. A bounded administrative reconciliation job may query missing message status without exposing credentials to a browser.

## Security and deliverability

- Store the Brevo API key in protected server configuration outside every public directory. Never commit it, return it to JavaScript, include it in a support package, or write it to a log.
- Use a dedicated least-purpose Brevo key for production and a different key for testing. Rotate keys through an audited Admin operation and verify that the previous key is rejected.
- Authenticate the selected sender domain in Brevo and publish the account-specific Brevo code, DKIM, and DMARC DNS records before production sending.
- Keep the sender address and reply-to address configurable and verified. Support mail replies continue through the controlled support workflow rather than exposing an administrator mailbox credential.
- Escape and validate every subject and parameter, prevent header injection, restrict links to trusted HTTPS origins, and prohibit secrets in query strings.
- Apply Admin Portal role checks, MFA, reauthentication, CSRF protection, confirmation, audience preview, consent preview, and an immutable audit entry to template activation, campaign approval, manual sends, credential rotation, and suppression overrides.

## Admin Portal controls

- Connection status without revealing the API key.
- Verified sender/domain state and last successful test.
- Configurable provider limit, automated limit, service reserve, quota time zone, quiet hours, and frequency caps.
- Template registry, preview, test send, activation, retirement, and version history.
- Queue status, message search, retry/cancel controls, failure reasons, and dead-letter review.
- Marketing audience preview with eligible, suppressed, unverified, duplicate, and over-quota counts before approval.
- Delivery, bounce, complaint, unsubscribe, and conversion summaries with clear disclosure that Free-plan statistics are limited and open/click measurements are not exact measures of customer interest.
- Emergency stop for all delivery and a separate marketing-only pause.

## Validation requirements

- Sandbox and test-recipient coverage must prove no real customer is contacted during development.
- Quota tests enqueue more than 300 messages and confirm the configured cap, service reserve, priority order, next-day continuation, cancellation, and no message loss.
- Retry tests cover `429`, `5xx`, timeouts before and after provider acceptance, malformed responses, and duplicate webhook events.
- Consent tests cover opt-in, withdrawal, unsubscribe, complaint, hard bounce, global suppression, required-service exceptions, and consent changes while mail is queued.
- Security tests confirm the API key and webhook credential never appear in repository files, browser responses, URLs, logs, exports, analytics, or desktop packages.
- Deliverability acceptance requires the sender domain to report authenticated and a real test message to pass DKIM and DMARC inspection before production campaigns are enabled.

## Owner setup required before implementation completion

1. Create a dedicated Brevo API key for the production backend. Do not paste it into GitHub, Codex chat, a source file, or browser JavaScript.
2. Choose the sender address and reply-to address under a domain controlled by EPCOM Ltd.
3. Add the exact Brevo code, DKIM, and DMARC values shown in the Brevo account to the domain DNS provider.
4. Create or approve the initial transactional and marketing templates in Brevo.
5. Confirm the Brevo account time zone used for the daily 300-send reset.
6. Provide final consent, unsubscribe, privacy, and business-identification language for legal review.
