# v0.3.47 Five-Day Promotional Trial design

## Customer experience

The License screen replaces promotional-key entry with a server-authorized evaluation flow:

1. The application checks eligibility with its existing protected installation identity.
2. An eligible verified customer selects Lite, Pro, or Enterprise.
3. **Start Five-Day Trial** sends an idempotent request to the licensing backend.
4. The backend atomically records the one-time promotion, signs the entitlement, and returns it only to the authenticated installation.
5. The application validates and stores the entitlement, then unlocks the selected edition immediately.

The active view identifies the evaluated edition, activation and expiration timestamps, live remaining time, and a purchase action. Expiration restores the prior permanent tier automatically. Customers never see, copy, or paste the promotional entitlement.

## Eligibility and lifecycle

The backend is authoritative for every lifecycle state:

- **Eligible:** verified active customer with an enrolled Customer Portal account and no prior promotion claim.
- **Verification required:** the installation is registered, but its customer record is not verified or has no Customer Portal account.
- **Active:** a promotion exists and its server expiration is in the future.
- **Used:** a promotion was already consumed, including after expiration.
- **Not applicable:** Enterprise already includes the highest available edition.

Trial customers may evaluate Lite, Pro, or Enterprise. Lite customers may evaluate Pro or Enterprise. Pro customers may evaluate Enterprise. A permanent Enterprise license is not offered a lower edition.

The evaluation lasts five consecutive 24-hour periods from the server-issued timestamp. The signed entitlement contains the prior tier, granted tier, subject, issue time, and expiration time. Local rollback detection pauses promotional access if the Windows clock moves backward beyond the allowed tolerance.

## Identity, one-use enforcement, and retries

- The desktop authenticates with the random installation token already stored in the protected application-data area.
- A request is bound to the authenticated installation and its canonical customer and permanent license, when present.
- The database permanently records unique claims for the customer, Customer Portal account, permanent license, and installation.
- Customer and account claims prevent reinstalling, deleting local files, changing editions, or using another registered installation from receiving a second promotion.
- An administrator may create one audited, single-use exception after reviewing the customer.
- The application stores a non-secret request UUID before contacting the server. A retry reuses that UUID.
- The backend stores the issued entitlement encrypted at rest so an authenticated retry can recover the same entitlement instead of consuming another promotion.
- Database uniqueness constraints and transactions reject concurrent or cross-edition duplicate requests.

## Security and privacy boundaries

- Entitlements are signed with the existing vendor private key on the protected server and verified with the public key in the desktop application.
- The signing key, database credentials, encryption key, and installation token are never returned to the browser UI or written to diagnostics.
- The recoverable entitlement is protected with AES-256-GCM at rest; its SHA-256 fingerprint remains the uniqueness key.
- The endpoint is POST-only, HTTPS-only in the desktop configuration, installation-authenticated, rate-limited, and returns generic server failures.
- Customer-visible failures omit server HTML, stack traces, credentials, database details, and transport diagnostics.
- No receipt content, raw print data, screenshots, printer payloads, or hardware identifiers are collected by this flow.
- The existing customer email verification and Customer Portal enrollment remain the ownership proof; email equality alone does not authorize a promotion.

## Failure and recovery behavior

- An offline or unreachable server leaves the current license unchanged and shows a customer-friendly retry message.
- A server rejection leaves the current license unchanged and preserves no activation key in the UI.
- A network interruption after server issuance reuses the saved request UUID and retrieves the same encrypted entitlement.
- A local persistence failure stops before requesting a promotion when the retry record cannot be saved.
- A corrupt, invalid, mismatched, future-dated, or expired entitlement is rejected by the desktop verifier.
- Expiration, clock rollback, and invalid records never damage the permanent activation key.

## Verification

- Automated desktop coverage verifies idempotent retry, immediate activation, clock rollback, expiration metadata, and customer-friendly network failure handling.
- PHP contracts verify endpoint authentication, rate limiting, one-use claims, encrypted recovery fields, exact used-state wording, and the absence of desktop signing credentials.
- Production UI validation covers edition selection, keyboard-accessible radio semantics, active-state content, and responsive single-column presentation.
- Release validation includes the complete .NET and PHP regression suites, PHP syntax checks, NuGet and package-manager vulnerability audits, installer packaging, checksum verification, and tracked-diff secret scanning.
