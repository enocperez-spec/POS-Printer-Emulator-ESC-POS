# v0.3.44 self-service commerce design

## Product boundary

v0.3.44 lets a verified Customer Portal account renew optional annual Application Maintenance and Support, permanently upgrade an owned license, or start one eligible five-day promotional paid-edition trial. A permanent license is never represented as a subscription. Ending maintenance removes future updates and assisted support only; ending a promotion restores the entitlement that existed before the promotion.

## Trust boundaries

- The Customer Portal authenticates the customer, enforces ownership, requires recent password reauthentication, and creates an opaque checkout intent.
- The Admin Portal database owns checkout intent, customer/license ownership, entitlement transitions, promotion eligibility, prior-entitlement snapshots, and immutable audit evidence.
- The Buy service owns current server-side prices and PayPal credentials. It resolves an opaque checkout token through an authenticated Admin service request, creates and captures the PayPal order, and reports the verified capture for idempotent fulfillment.
- Browser code receives only display-safe order details, a PayPal order identifier, and opaque references. It never receives PayPal secrets, service credentials, activation-signing keys, full activation keys, database credentials, or arbitrary prices supplied by the browser.
- The desktop accepts a temporary promotion only through a signed, expiring entitlement bound to the expected license or installation. Local clock rollback invalidates the temporary entitlement instead of extending it.

## Checkout sequence

1. The signed-in customer selects maintenance renewal or an allowed upward tier change.
2. The portal requires a password confirmation no more than five minutes old.
3. The Admin service locks the owned license, rechecks control and maintenance state, creates an expiring checkout intent, and stores only a digest of the browser token.
4. The Buy service resolves the opaque token through its authenticated service channel, loads the current server-controlled price, and displays the exact product, tier, amount, currency, maintenance effect, and refund/support terms.
5. PayPal creates and captures a one-time order with an idempotency key bound to the internal intent.
6. The Buy service verifies provider status, capture identifier, amount, currency, and capture time before reporting fulfillment.
7. The Admin service applies the transition once. Repeated capture callbacks return the existing result and cannot issue duplicate maintenance or a second replacement license.
8. The portal shows the completed purchase in customer history. Activation material remains protected and follows the established replacement-key delivery and desktop refresh path.

## Promotional trial sequence

1. Eligibility is evaluated from canonical customer, account, license, installation, and prior-promotion records.
2. Each verified customer may normally receive one promotion. An administrator exception requires a reason and its own audit event.
3. Starting a promotion records the prior tier, the granted tier, the bound license or installation, start and expiration timestamps, and the eligibility decision.
4. A signed five-day entitlement is available to the bound desktop installation. No payment method is requested and no order is created.
5. At expiration, the server marks the promotion expired and the desktop drops the signed temporary entitlement. Its permanent or Trial entitlement remains unchanged.
6. Offline clients may use the signed entitlement only until its embedded expiration. A persisted highest-observed time prevents clock rollback from extending access.

## Failure and reversal rules

- Expired checkout tokens are not renewed implicitly; the customer must review a fresh current price.
- A failed or canceled payment leaves the permanent license and maintenance dates unchanged.
- Duplicate PayPal callbacks return the original fulfillment result.
- Refunds and chargebacks create auditable review events. A permanent downgrade or revocation requires the existing administrative confirmation workflow; it is not silently performed by a browser callback.
- If email delivery fails after fulfillment, the fulfilled entitlement remains intact and delivery can be retried without issuing a duplicate.
- A promotion start is atomic. If signed entitlement creation fails, the promotion record is rolled back.

## Visual system

The portal extends the v0.3.43 deep-navy navigation and true-white workspace. The new **Plans & maintenance** page uses one open content rail, a compact edition comparison table, a maintenance renewal band, a promotional-trial band, and a right-side order review drawer. It preserves the existing portal type, spacing, borders, focus treatment, and mobile collapse. Pro is limited to two listeners; Enterprise displays a recommended maximum of fifteen.

## Release verification

- Server-side price and ownership tests for every tier and action.
- Duplicate create, capture, callback, and fulfillment tests.
- Payment mismatch, stale intent, failed payment, refund, and chargeback tests.
- Promotion single-use, administrator exception, expiration, restoration, offline expiry, and clock-rollback tests.
- Cross-customer object authorization and rate-limit tests.
- Desktop signed-entitlement codec and effective-tier tests.
- Desktop and mobile browser verification against the approved concept.
