# v0.3.43 Secure Customer Portal MVP

## Product boundary

The Customer Portal gives a verified customer self-service access to records that already belong to the canonical customer ID created in v0.3.42. It does not issue activation keys, change a license tier, process payments, or grant promotional entitlements. Those commercial mutations remain in v0.3.44.

The portal is deployed at `userportal.posprinteremulator.com` with its own document root and protected configuration. It shares the administrative database through a least-privilege database account and sends support submissions to the Admin Portal through an authenticated server-to-server endpoint. Browser code never receives database, GitHub, email-provider, license-signing, or service API credentials.

## Authentication

- Enrollment and password recovery return the same response whether or not an email matches.
- Verification and reset links use random 256-bit tokens. Only SHA-256 token digests are stored, links expire after 30 minutes, and each link is single-use.
- Passwords use PHP Argon2id with the runtime's current secure defaults and must contain at least 12 characters, uppercase, lowercase, and a number.
- Failed login attempts create a 15-minute account lock after five failures. Source and workflow rate limits persist in the database.
- Customer sessions use `Secure`, `HttpOnly`, `SameSite=Strict` cookies, strict session mode, a 30-minute idle limit, a 12-hour absolute limit, ten-minute rotation, user-agent binding, database revocation, and an account session revision.
- Optional TOTP accepts one adjacent 30-second clock step. Secrets are encrypted with AES-256-GCM using a portal-only protected key. Eight recovery codes are displayed once and stored only as SHA-256 digests.
- Password reset increments the session revision and revokes all active sessions.

## Authorization and privacy

Every customer query includes the authenticated canonical `customer_id`. Route parameters never select a license, installation, support request, purchase, consent, or reply without that ownership predicate.

The portal displays masked activation-key endings only. Account export intentionally excludes complete activation keys, diagnostics, attachments, raw receipts, print data, screenshots, credentials, and private-network addresses.

Profile name updates, consent decisions, sign-in, enrollment, password reset, MFA enrollment, support actions, data export, device deactivation, and account closure add customer audit events. Consent remains append-only.

## Sensitive actions

Computer deactivation and portal closure require password reauthentication no more than five minutes earlier. Deactivation:

- locks the installation row,
- verifies canonical ownership,
- rejects an already deactivated installation,
- allows at most two deactivations per customer in 24 hours,
- records an immutable device action and customer event,
- causes subsequent telemetry from that installation to be acknowledged without updating activity.

Closing portal access marks the customer record closed, adds an email suppression, increments the session revision, and signs the customer out. It does not cancel, revoke, delete, or convert a permanent software license.

## Support

Customer subjects, descriptions, and validated attachments remain in the protected administrative database. The Customer Portal sends only the opaque support reference to the Admin Portal backend. The Admin Portal creates a minimal public GitHub issue containing the request type and opaque reference; subjects, contact information, descriptions, diagnostics, replies, and attachments remain private.

Attachments are limited to one PNG, JPEG, TXT, PDF, or ZIP file no larger than 2 MB. Support replies enforce customer ownership and are visible in both the Customer Portal and Admin Portal.

## Visual system

The accepted design uses the product's deep navy navigation, true-white content surfaces, blue/cyan actions, green status, restrained warnings, and table/list-based data. It avoids dashboard metric decoration and never turns customer data into fake charts.

Desktop uses a quiet fixed navigation rail and an open content canvas. Mobile moves the rail behind a keyboard-accessible menu while retaining the same page order and task labels.

## Production configuration

The protected portal configuration contains:

- a least-privilege database connection,
- the canonical HTTPS base URL,
- an independent random 32-byte AES key encoded as Base64,
- a server-side Admin Portal support handoff token,
- a mail transport and verified sender.

Real values are never stored in Git, logs, HTML, JavaScript, URLs other than single-use verification/reset tokens, or downloadable support material.
