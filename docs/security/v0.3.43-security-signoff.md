## Completed controls

- Generic enrollment and recovery responses.
- Single eligible identity requirement; ambiguous matching emails are not merged or exposed.
- Argon2id passwords, persistent throttling, lockout, short reset-token lifetime, and digest-only token storage.
- Secure HttpOnly SameSite cookies, session rotation, idle and absolute expiration, user-agent binding, server-side revocation, CSRF protection, CSP, no-store caching, HSTS, and clickjacking protection.
- Customer-ID ownership checks on licenses, computers, purchases, support requests, replies, consents, and events.
- Masked activation-key endings only; no full activation key is selected by portal queries.
- AES-256-GCM protection for TOTP secrets with a portal-only key retained through Windows DPAPI for safe redeployment.
- Reauthentication, row locking, cooldowns, and audit evidence for old-computer deactivation.
- Strict upload size and MIME allowlists; private attachments never enter public GitHub issues.
- Public GitHub issues use an opaque support reference and request type only. Customer subject, contact, description, replies, diagnostics, and attachments remain private.
- Protected server-to-server support and migration endpoints authenticate with a hashed bearer credential.
- Portal configuration and deployment secrets are excluded from normal publishing and from repository source.

## Required release verification

1. Apply the authenticated v0.3.43 schema migration to a production-equivalent database.
2. Deploy the portal with its protected configuration and verified mail sender.
3. Confirm HTTPS redirect, TLS certificate, HSTS, CSP, cookie flags, and cache headers at the public hostname.
4. Exercise enrollment, generic recovery, login throttling, MFA and recovery codes, session revocation, ownership isolation, data export, support submission and retry, and device-deactivation cooldowns with dedicated test customers.
5. Authenticode-sign the self-contained v0.3.43 Windows installer, then install, upgrade, repair, and uninstall it and run a restricted-user smoke test.
6. Verify the published SHA-256 checksum against the signed installer.
7. Perform an independent release review and change the decision to `APPROVED` only when no critical or high-severity issue remains.
