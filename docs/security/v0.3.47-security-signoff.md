## Verified controls

- The desktop sends promotional requests only to the fixed HTTPS Admin Portal origin.
- Requests use the existing random installation identifier and protected installation token; neither value is exposed in the viewer.
- The backend resolves the installation to an active canonical customer, verified email, Customer Portal account, and permanent license before issuing an entitlement.
- Trial eligibility and the one-use decision are server-side. Unique customer, account, license, and installation claims survive reinstall, local-file deletion, edition changes, and alternate registered installations.
- Entitlements are signed only on the backend. The desktop contains the verification key, not the vendor private key.
- Recoverable retry tokens are encrypted at rest with AES-256-GCM and additionally fingerprinted with SHA-256.
- Request UUIDs provide idempotency, and the backend returns the same entitlement only when the authenticated customer, subject, and requested tier match.
- Database transactions and unique claim constraints reject concurrent duplicate issuance.
- The endpoint is POST-only, rate-limited, no-store, and uses generic authentication and server failures.
- The desktop rejects invalid tiers, mismatched subjects or prior tiers, future-dated or expired entitlements, invalid signatures, and clock rollback.
- Offline, timeout, persistence, and server failures leave the permanent license unchanged and return customer-friendly messages without raw transport or server diagnostics.
- The viewer contains no promotional entitlement or manual-key field.
- No receipt data, raw print payload, screenshot, printer content, or hardware identifier is collected by the promotional flow.

## Verification evidence

- Idempotency coverage simulates a failed response followed by a retry and proves that the same request UUID is used and the returned entitlement activates immediately.
- Network-failure coverage proves that low-level transport details are not returned to the customer.
- License coverage verifies active timestamps, expiration behavior, prior-tier restoration, and clock-rollback protection.
- PHP contracts verify the authenticated endpoint, rate limit, one-use claims, encrypted recovery fields, exact already-used message, and lack of embedded signing credentials.
- Desktop and narrow-window UI review verified accessible edition radio semantics, one selected edition, a single clear start action, and responsive one-column layout.
- The release pipeline rebuilt the viewer, service, desktop host, updater, license tools, self-contained Windows runtime, and installer.
- Published candidate executables report file version `0.3.47.0`; bundled SQLite verification passed.
- The generated installer checksum matches an independent SHA-256 calculation.

## Residual risk acceptance

The Windows installer is not Authenticode-signed because a trusted code-signing certificate is not configured. Customers may receive a Windows SmartScreen warning. The official HTTPS download, separately published SHA-256 checksum, GitHub release assets, and in-application checksum verification remain mandatory integrity controls until production code signing is implemented.

The installer and backend rely on the existing protected application-data and server configuration controls. A compromised Windows administrator account or compromised production hosting account remains outside the protection offered by application-level encryption.

No critical or high-severity vulnerability was identified in the v0.3.47 release changes.
