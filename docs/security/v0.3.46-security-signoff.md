## Verified controls

- Window-placement preferences contain only normal-window bounds and a maximized/normal state; they contain no customer registration, license key, receipt content, printer data, credential, or network address.
- Persisted values are parsed with invariant numeric rules, constrained to minimum usable dimensions, and moved inside the current Windows work area before application.
- A minimized state is never restored, preventing an apparently missing application after restart.
- Corrupt, incomplete, off-screen, or disconnected-monitor placement data falls back to safe taskbar-aware centering.
- The feature does not add a network endpoint, privilege boundary, installer elevation path, upload, import format, or license-tier bypass.
- Existing update downloads continue to require trusted GitHub installer and checksum assets over HTTPS.
- Release metadata points only to the official HTTPS website and GitHub release.
- The tracked release diff contains no newly introduced credential, API key, private key, or password.
- NuGet and pnpm audits reported no known vulnerable packages in the reviewed release projects.

## Verification evidence

- Six focused placement-preference tests cover first use, normal placement, maximized placement, minimized-state rejection, disconnected-monitor recovery, and corrupt preference fallback.
- The complete desktop suite passed 181 of 181 tests.
- The release pipeline rebuilt the viewer, service, desktop host, updater, and self-contained runtime with zero warnings or errors.
- Published executable file versions are `0.3.46.0`; the bundled SQLite runtime verification passed before packaging.
- Inno Setup compiled the all-in-one Windows installer successfully using the validated product icon and task-specific installer banner.
- Every PHP contract suite passed, all 71 PHP files passed syntax validation, and the website publisher passed release synchronization plus all 27 canonical SEO checks.

## Residual risk acceptance

The Windows installer is not Authenticode-signed because a trusted code-signing certificate is not configured. Customers may receive a Windows SmartScreen warning. The official HTTPS download, separately published SHA-256 checksum, GitHub release assets, and in-application checksum verification remain mandatory integrity controls until production code signing is implemented.

No critical or high-severity vulnerability was identified in the v0.3.46 release changes.
