## Verified controls

- Brevo credentials, webhook authentication material, database credentials, and template configuration remain outside public directories, repository source, browser code, and desktop binaries.
- Every queued message records an idempotency key and is rechecked for verified ownership, consent, recipient hash, suppression, account state, and emergency-stop state immediately before delivery.
- Required service messages and optional marketing use separate policy paths, pause controls, frequency limits, and reserved quota.
- The Free-plan provider limit is enforced locally with conservative automated-message and reserved-service limits; excess work remains durably queued for a later quota window.
- Hard bounces, complaints, unsubscribes, and administrative suppressions stop future eligible delivery.
- Webhook ingestion authenticates requests, validates payloads, deduplicates events, and avoids logging credentials or message contents.
- Admin communications mutations require authorization, CSRF protection, confirmation, audit evidence, and recent two-factor verification where required.
- Analytics and template parameters exclude activation keys, credentials, receipt text, raw print data, private listener addresses, logs, and personal printed content.
- Provider delivery and marketing default to paused when protected configuration is absent.
- Release metadata points only to HTTPS website and official GitHub destinations, and the published installer has an independently verified checksum.

## Deployment limitations and required activation

- The release is intentionally deployed with Brevo delivery paused because protected REST API credentials, verified sender identity, and approved transactional template IDs are not present in this workspace.
- Do not reuse the previously disclosed SMTP credential. Rotate it in Brevo if it has not already been revoked.
- Before enabling live delivery, install a newly generated restricted Brevo REST API key in protected server configuration, verify the sender/domain, configure approved template IDs, apply the authenticated communications migration, send only to a controlled test allowlist, verify webhook authentication and suppression handling, and then remove the emergency stop.
- The Windows installer is not Authenticode-signed because no trusted code-signing certificate is available. The official HTTPS download and published SHA-256 checksum are required integrity controls until signing is implemented.

## Residual risk acceptance

No critical or high-severity vulnerability was found in the v0.3.45 release changes during the implementation review and automated security contracts. The release owner accepts the temporary SmartScreen/publisher-reputation limitation of an unsigned installer and the operational limitation that communications remain inactive until the protected production activation checklist is completed.
